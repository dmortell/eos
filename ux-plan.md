# UX Plan — Unified Workspace

## Decided direction

The current tool-per-route layout has two real problems:

1. **Tool switching is jarring.** Each tool is a separate route with its own selector for floor/room/row/rack, its own pan/zoom state, and its own sidebar. The dominant switch in practice is Frames → Racks → back to Frames (to fix a device or panel mid-wiring), and every switch costs a route change, a re-subscribe, and a lost viewport.
2. **Patching's identifiers don't match what's physically on the panels.** Patching currently uses rack U / port#. The labels printed and taped to the ports come from Frames. Field teams read the sticker on the port and try to find it in the patching docs — and the two don't agree. This is a data-integrity bug, not a UX preference.

The fix is a unified workspace: one route, a shared tree navigator, view-mode buttons that swap how the selected entity is rendered, and editing surfaces split by content shape (right panel for properties/library, bottom panel for tabular data, floating for transient wizards). Frames + Patching merge into one tool with two render modes, backed by one data model.

## Tool consolidation

| Today | Tomorrow |
|---|---|
| Racks | **Racks view** — row elevation overview |
| Frames | **Frames view** — port-level detail with full labels |
| Patching | **Patching view** — row-level cabling overview with compact labels |
| Outlets | **Outlets view** — floorplan with high/low outlets and trunks |
| Risers | **Risers view** — building elevation with inter-floor connections |

These become view modes inside one workspace route, not separate pages. Frames and Patching share one data model — port labels live in Frames data, Patching reads them, no duplicate identifiers.

## Label model

**Frames port labels are canonical.** Patching reads them. Outlets on the floorplan and the panel-side ports they connect to share the same label string by construction — one outlet ↔ one panel port ↔ one label.

### Sync rules

- A label exists when assigned to a port in Frames. Until then the port is **unlabeled**.
- **Floorplan outlets are multi-port.** The outlet label does NOT include the port number. A 2-port outlet labeled `33-001` exposes ports `33-001.A` and `33-001.B` on the patch frame. Plotting an outlet on a floorplan makes its ports available for assignment in Frames; each port label is generated by appending the port suffix to the outlet label.
- Renaming a label in any view updates the single source. The patch list, the outlet annotation, and the rack-elevation port tag all reflect it live.
- Deleting a label requires the port to be unpatched and unlinked from any outlet (soft validation; warn and confirm).

### Label format varies per client

Clients differ on:
- **Separators** — hyphens vs periods (`F1-Z2-001-A1` vs `F1.Z2.001.A1`).
- **Components included** — some clients want zone or room in the label, some don't.
- **Port suffix style** — alpha (`.A`, `.B`) vs numeric (`.1`, `.2`).

A label-format settings tool is planned (see plan item 10). Until then, labels follow the current `FF.Z.NNN-SPP` convention with `-H` for high-level.

### Port-level patching gate

Patching unlabeled ports is disabled at the port level, not the tool level:

- Unlabeled ports render as `—` (or compact U/port in grey) and reject patch connections.
- Clicking an unlabeled port in Patching view opens an inline label-entry field at the port. Type → enter → port is now labeled and patchable in the same gesture.
- The right-panel inspector exposes the same label field for the selected port.

Why this over U/port-as-temp-label: even with "temp" UI markers, U/port labels leak into screenshots, the live patch list IT teams consult, and printed deliverables. One canonical label is the whole point of the merge.

Why this over a global tool disable: you'd be unable to patch any port until every port had a label, which doesn't match how engineers actually work.

## Label rendering — experiment behind settings

The original Frames/Patching split existed because labels readable at panel-detail zoom are unreadable at row-overview zoom. Several workarounds exist; we don't yet know which one wins. Ship them as **stackable settings** so we can A/B in a real session.

```ts
// per-user, localStorage
labelRendering = {
  lod: true,             // shrink / abbreviate labels at low zoom (option 1)
  tooltip: true,         // full label in tooltip on hover (option 2)
  hoverMagnifier: false, // magnifying-glass lens around cursor enlarges nearby labels (option 6)
  inspectorAlways: true, // right panel always shows full label of selection (option 3)
  splitViewModes: false, // distinct Frames vs Patching render with different default label sizes (option 4)
}
```

A small "Label rendering" section in the workspace settings exposes these toggles. Defaults to `lod + tooltip + inspectorAlways` (the cheapest combination that should "just work"). `hoverMagnifier` is the experimental fun one; `splitViewModes` is the closest-to-current-behaviour fallback.

Printing rack elevations and panel-label sheets for the drawings/packaging tools is a separate render path — always full labels at fixed scale, no LOD. Print is a deliverable; on-screen is interactive.

### Tree navigator

Shape:
```
Project
├── Floors
│   └── F1 / F2 / …
│       ├── Outlets (high)
│       ├── Outlets (low)
│       └── Trunks
├── Building
│   └── Risers
└── Server rooms
    └── SR-01
        └── Row A
            └── Rack-01
                ├── Panel 1
                └── Panel 2
```

- **Shows only what exists in the project.** No placeholders for unbuilt floors or empty racks; creation happens through normal "add" actions, not by navigating to a stub.
- Selecting any node updates the canvas + view strip. Selecting a Panel sets the viewport so the canvas frames that panel; it does not change the view component.
- Single shared selector — every tool gets the same nav UX.

### View strip

Available views depend on the tree selection:

| Tree selection | View strip |
|---|---|
| Floor | Outlets (high) · Outlets (low) · Trunks · Fill rate |
| Server room | Rooms overview |
| Row | Row elevation |
| Rack | Elevation · Frames · Patching |
| Panel | Frames · Patching (both zoomed to the panel) |
| Building → Risers | Risers |

### Editing surfaces

- **Right panel — properties & libraries.** Selection properties (port label, device specs), drag-source libraries (devices, panels, outlets). Vertical content.
- **Bottom panel — tabular data.** Live patch list, label assignment grids, port-by-port edit tables. Collapsible like a terminal pane. The patch list lives here as a live view; same data model as the canvas, no separate "patch list page."
- **Floating windows — transient wizards only.** Bulk-assign labels from floorplan outlets; import devices from CSV; print/export pickers. Open → complete → close.

Reserving floating windows for transient wizards avoids the worst trap of floating UI (users losing the panel off-screen).

## Patch list as live view

The bottom-panel patch list reflects current cable connections in real time. No "save and reopen" cycle. Many IT teams won't consult it after first install — that's fine; for the engineers building the patch plan it's the active surface.

Printing/exporting the patch list for the drawings/packages tools is a separate render path: snapshot of current connections at fixed format, served by the same data model.

## Plan of attack

1. **Stable sidebar shell + workspace route** in the `[pid]` layout. Tools become components, not pages. Existing tool routes redirect to the workspace with the right tree selection. *(Medium.)*
2. **Tree navigator** populated from existing Firestore collections (frames/{pid}, racks/{pid}, files/{fileId}, risers data). Only shows what exists. *(Small once #1 lands.)*
3. **URL viewport encoding** — `?node=<tree-path>&v=x,y,z`. Read on mount, write on pan/zoom. Tabs/local state stay in localStorage. *(Small.)*
4. **Shared `RackElevation` component** extracted from Racks. Used by Racks view, Frames view, and Patching view. Devices + panels in all three; the views overlay different interactive surfaces on top. *(Medium — the load-bearing extract.)*
5. **Label model unification.** Patching reads labels from Frames data; remove Patching's U/port label representation. Port-level patching gate with inline label entry. *(Small once Frames data is the source.)*
6. **Label rendering settings panel.** Stackable toggles (`lod`, `tooltip`, `hoverMagnifier`, `inspectorAlways`, `splitViewModes`). Implement each as an independently-toggleable behaviour. *(Small per toggle; ship over a few days.)*
7. **Bottom panel with live patch list.** Reads from the same cabling data; collapsible. *(Small.)*
8. **Right panel inspector** — properties for current selection, library tabs for device/panel/outlet drag sources. *(Medium.)*
9. **Risers as a view** in the workspace. Same shell, different canvas. *(Small structural; the canvas itself is its own work.)*
10. **Label-format settings tool.** Per-project settings for separator, included components (zone/room), port suffix style (alpha vs numeric). Label rendering reads from settings; existing labels can be re-rendered without altering stored data if we store labels as structured parts rather than pre-joined strings — decide storage shape during implementation. *(Medium; can ship after #5.)*

Test data is sparse and disposable — no migration needed for the Patching label change.

## Considered and parked

- **Split-view panes (Option B from earlier drafts).** Worth revisiting after the workspace shell exists; trivial to add then. Not needed if in-place view switching feels good.
- **Layers-instead-of-tools (Option D).** Conceptually clean, but mixes view rendering with editing-mode semantics in ways that don't map well to how engineers think about the work. North star at best.
- **Per-Firestore tab persistence.** Decided against — tabs are per-browser (localStorage). Per-device sync isn't worth the write volume.

## Decided constraints (recap)

- Tabs / workspace state — per-browser (localStorage).
- URL encodes only the active selection + viewport for share-link sharing. Drawings/packages embedding will consume the same URL shape.
- Frames port labels are canonical; Patching and Outlets read them.
- Port-level patching gate; no temp U/port labels in the patch list.
- Patch list is a live view, not a saved deliverable.
- Tree shows only what exists in the project.
- Label rendering: ship multiple workarounds behind settings, pick a default once we've tried them.
